## Purpose

This note describes the intended implementation shape for the first
remote operator bootstrap API.

The goal is to let trusted operator automation or browser-based operator tools
perform onboarding and device bootstrap without reproducing admin CLI logic,
touching server storage directly, or widening the operator surface beyond what
ADR-0002 and ADR-0006 allow.

## Current State

Today the server already has:

- user REST auth
- operator HTTP auth for `/admin/*`
- local admin CLI flows for user, sync identity, device, backup, and recovery
- a shared canonical sync identity service
- a shared device provisioning service

What it does not yet have is a remote-friendly operator bootstrap flow.

Current gaps:

1. no operator HTTP endpoint that can bootstrap a user + canonical sync
   identity + real device in one flow
2. no operator HTTP surface for canonical sync identity lifecycle
3. no remote operator device lifecycle for a target user
4. no OpenAPI-emitted operator surface for generated operator clients

## ADR-0002 Position

The operator bootstrap slice is compatible with ADR-0002 only if it lands as a **thin operator
surface over narrow services**.

That means:

- request parsing happens at the HTTP boundary
- user/bootstrap orchestration lives in a small service
- canonical sync identity rules stay in the sync identity service
- device provisioning rules stay in the device service
- backup/restore and break-glass filesystem repair remain CLI-first

What ADR-0002 does **not** support here:

- putting all bootstrap logic directly into handlers
- re-implementing CLI logic in external operator tooling
- pushing restore/repair workflows into remote HTTP just because they are
  convenient

## Recommended First Slice

The first implementation slice should add one operator-authenticated endpoint:

- `POST /admin/bootstrap/user-device`

This endpoint should perform one narrow orchestration:

1. resolve an existing target user or create one if explicitly allowed
2. ensure the user has a canonical sync identity
3. provision a real device for that user
4. return the one-time credential payload needed by the target client and the
   operator workflow delivering it

This should be the smallest server-side surface that unblocks the first remote
onboarding flow built on top of server-owned bootstrap primitives.

## First API Shape

The bootstrap request should be typed and explicit.

Likely request fields:

- `userId` or other stable server-side user identifier when known
- `username` or equivalent create-if-missing identity input
- `createUserIfMissing`
- `deviceName`
- `bootstrapRequestId`
- optional `publicServerUrlOverride`

Likely response fields:

- `userId`
- `username`
- `canonicalClientId`
- `deviceClientId`
- `encryptionSecret`
- `serverUrl`
- client-ready sync/bootstrap config values

The exact field names can settle later, but the important shape is:

- one typed bootstrap request
- one typed bootstrap response
- no opaque JSON blob handoff between layers

## Idempotency Rules

The bootstrap flow should be safe to retry.

Expected semantics:

- existing user + existing canonical sync identity is not an error
- retry after network loss should not force operator cleanup
- device creation should remain explicit and non-accidental

Recommended behaviour:

- "ensure canonical sync identity" is idempotent
- device creation for bootstrap is keyed by a required idempotency token /
  external bootstrap correlation id

The implementation should **not** use an "always create a fresh device on
retry" rule for the bootstrap endpoint.

That easy path is unsafe because a timeout after commit can leave:

- an orphaned device record
- a one-time credential that was never delivered to the operator workflow or
  target client
- no safe way to distinguish an intentional second bootstrap from a retry

### Required Bootstrap Idempotency Input

The bootstrap request should require:

- `bootstrapRequestId`

This value is caller-generated by the operator workflow and is the stable key
for retry semantics.

The server should treat a repeated bootstrap request with the same
`bootstrapRequestId` as the **same logical bootstrap attempt**, not as a
request to create a second device.

### Required Retry Semantics

Before implementation starts, the endpoint contract must define one concrete
retry policy.

Recommended policy:

1. the server records bootstrap attempt state keyed by `bootstrapRequestId`
2. if the request reaches the point where device credentials are created but
   the attempt is not yet acknowledged as delivered, a retry with the same
   `bootstrapRequestId` replaces that incomplete bootstrap device with a fresh
   one-time credential
3. once the attempt is marked delivered/complete, further retries with the
   same `bootstrapRequestId` return the final stable bootstrap result without
   creating another device

If a different retry policy is chosen, it must still answer the same core
question explicitly:

- how does the system recover when the server committed bootstrap state but the
  operator workflow never received the one-time secret?

### Required Acknowledgement Mechanism

The retry model requires a concrete acknowledgement signal.

Recommended primary mechanism:

- `POST /admin/bootstrap/{bootstrapRequestId}/ack`

Semantics:

1. the bootstrap endpoint creates or replaces an incomplete bootstrap device
   and returns the one-time credential payload
2. the operator workflow durably records or forwards that payload through its
   own delivery path
3. the operator workflow calls the ACK endpoint to mark that bootstrap attempt as
   delivered/complete
4. only after that ACK is the server allowed to treat later retries for the
   same `bootstrapRequestId` as a stable replay of the completed result

This should be the primary completion signal for the first implementation
slice.

The first successful device sync may later be recorded as an additional
observability signal, but it should also act as an **implicit completion
signal** for retry safety.

That means:

- explicit operator ACK is the primary protocol completion signal
- first successful device sync marks the bootstrap device as complete if the
  explicit ACK was missed

This avoids the handoff race where:

1. the operator workflow gives the credential to the client
2. the operator workflow crashes before sending the ACK
3. the client has already started using the credential
4. a retry would otherwise revoke a credential that is now live

The implementation should therefore treat either of these as sufficient to
move the bootstrap device out of the incomplete state:

- explicit `POST /admin/bootstrap/{bootstrapRequestId}/ack`
- first successful authenticated sync from that device

## Public URL Requirement

Bootstrap responses must return the **externally reachable** server URL, not
the internal bind address.

That means the implementation needs a configured public base URL, for example:

- `[server].public_base_url`
- `CMDOCK_PUBLIC_BASE_URL`

The bootstrap flow must not derive client-facing URLs from:

- `server.host`
- `server.port`

because those describe the local listener, not the reverse-proxied public
address used by first-party clients.

## Service Boundary

The intended server-side layering is:

1. operator HTTP handler
   - auth
   - typed request parsing
   - response mapping

2. bootstrap orchestration service
   - resolve/create user
   - ensure canonical sync identity
   - provision device
   - coordinate rollback / atomicity policy

3. existing narrow services
   - sync identity service
   - device provisioning service
   - recovery service where needed

This keeps the bootstrap flow aligned with ADR-0002:

- one new operator feature should not require edits across unrelated domains
- handlers stay thin
- business rules live behind small service seams

## Failure Handling Requirement

This is a hard requirement.

Bootstrap must not leave behind half-created operator state when it fails after
the one-time credential should have been returned.

At minimum the implementation must address:

- device row created but credential response never delivered
- canonical sync identity created but device creation failed
- unknown user IDs being taken offline or otherwise mutating runtime state

This is not just a database transaction problem.

A SQL transaction can protect:

- user creation
- sync identity creation
- device row creation

It cannot by itself solve:

- HTTP response loss after commit
- operator-workflow uncertainty about whether the one-time secret was delivered

### Required Concrete Failure Strategy

Before implementation starts, this note requires a concrete bootstrap cleanup
strategy rather than generic "rollback where possible" language.

Recommended direction:

1. persist bootstrap-attempt state keyed by `bootstrapRequestId`
2. track whether the one-time credential was only generated, returned, and
   acknowledged as consumed
3. on retry of an incomplete attempt, revoke/delete the incomplete bootstrap
   device and mint a new one-time credential
4. add TTL-based cleanup for abandoned incomplete bootstrap attempts
5. keep canonical sync identity creation idempotent so retries do not multiply
   sync identities

If a lighter-weight design is chosen, it still needs explicit answers for:

- where incomplete bootstrap state is recorded
- how abandoned device rows are identified
- how a retry safely replaces or resumes an incomplete bootstrap

### Recommended State Storage Shape

Avoid introducing a separate bootstrap-attempt domain table unless the device
row cannot carry the needed state cleanly.

Preferred first implementation:

- store `bootstrapRequestId` on the bootstrap-created device row
- store a bootstrap status on that same row, such as:
  - `pending_delivery`
  - `acknowledged`
  - `abandoned`
- store TTL / expiry metadata on that row

This keeps the first slice aligned with ADR-0002 by reusing the existing device
domain rather than introducing a parallel bootstrap-attempt subsystem too
early.

If later operator-automation requirements outgrow that shape, the state can be split
out deliberately then.

### Required Auth and Sync Rule for Bootstrap Devices

The bootstrap status must participate in device auth decisions explicitly.

Recommended rule:

- `pending_delivery` devices are allowed to authenticate and sync
- the first successful sync transitions the device to the completed state if it
  was not already explicitly ACKed
- `abandoned` devices are rejected by sync auth
- normal active devices are unaffected

The sync/auth path must not silently ignore bootstrap status fields. If the
bootstrap status lives on the main device row, the implementation must audit:

- device-auth lookup
- revoked-device rejection
- sync freshness/device state transitions

so bootstrap-only states do not leak into normal lifecycle handling by
accident.

## Follow-On Operator APIs

After the first bootstrap endpoint, the next operator HTTP candidates are:

1. canonical sync identity ensure/show
2. operator device lifecycle for a target user

Likely shape:

- `POST /admin/user/{id}/sync-identity/ensure`
- `GET /admin/user/{id}/sync-identity`
- `GET /admin/user/{id}/devices`
- `POST /admin/user/{id}/devices`
- `PATCH /admin/user/{id}/devices/{client_id}`
- `POST /admin/user/{id}/devices/{client_id}/revoke`
- `POST /admin/user/{id}/devices/{client_id}/unrevoke`
- `DELETE /admin/user/{id}/devices/{client_id}`

These should build on the same service boundaries as the initial bootstrap
endpoint.

## User Resolution Rules

The bootstrap request must not rely on fuzzy identity resolution.

Before implementation starts, the contract should define strict rules for the
allowed request shapes.

Recommended allowed shapes:

1. `userId` only
   - resolve an existing user by server user ID
   - return `404` if not found

2. `username` only with `createUserIfMissing = false`
   - resolve an existing user by username
   - return `404` if not found

3. `username` only with `createUserIfMissing = true`
   - resolve by username if it already exists
   - otherwise create a new user with that username

4. `userId` and `username` together
   - only valid if both point to the same existing user
   - otherwise return `409 Conflict`

The endpoint should not silently prefer one identifier over the other.

Invalid combinations should fail deterministically, for example:

- missing both `userId` and `username` -> `400`
- `createUserIfMissing = true` with `userId` only -> `400`
- `userId` found but `username` mismatched -> `409`
- `username` exists for a different user than `userId` -> `409`

### Concurrent User Creation Rule

The `createUserIfMissing = true` flow must be race-safe.

Recommended rule:

1. enforce username uniqueness in the config store
2. when create-if-missing is requested, attempt create once inside normal
   uniqueness enforcement
3. if creation loses a uniqueness race, catch that specific uniqueness error,
   reload by username, and continue with
   the existing user
4. do not surface this normal race as `500`

This keeps the bootstrap service deterministic under concurrent onboarding
redemption or retried operator calls.

The implementation does not need a new abstract "bootstrap UPSERT" primitive,
but it must use database-level uniqueness plus explicit uniqueness-error
handling. A read-then-create sequence on its own is not sufficient.

### Request Consistency Rule

`bootstrapRequestId` must be bound to one logical bootstrap payload.

If the same `bootstrapRequestId` is retried with materially different request
parameters, for example:

- different target user
- different username
- different device name

the server should reject the retry with `409 Conflict` rather than silently
reinterpreting the bootstrap attempt.

This validation is required, not optional.

At minimum the server should compare the retried request against the original
attempt for:

- target user identity
- username input
- create-if-missing intent
- device name

The server must not permit a `bootstrapRequestId` created for one logical
bootstrap request to be reused for a different target or device.

### TTL Cleanup Rule

TTL cleanup for incomplete bootstrap devices should be implemented as a lazy
inline sweep, not as a new background worker in the first slice.

Recommended rule:

1. on each bootstrap create/retry request, sweep expired incomplete bootstrap
   devices for the relevant user or request scope
2. mark or delete those expired devices inline before proceeding
3. keep the logic local to bootstrap orchestration rather than introducing a
   general runtime cleanup subsystem

This keeps the first implementation aligned with ADR-0002 by avoiding a new
background lifecycle mechanism unless a real production need appears later.

## Non-Goals

This note does not propose:

- moving backup/restore into remote HTTP
- adding deployment-specific RBAC/team-admin semantics
- designing the full push notification hook
- adding billing/subscription projection to the server runtime
- exposing broad filesystem repair over HTTP

Those are separate concerns and should not be braided into the first bootstrap
API slice.

## Testing Requirements

The first implementation slice should add or update:

- bootstrap happy-path integration test
- bootstrap retry/idempotency test
- bootstrap retry after lost-response / incomplete-attempt simulation
- bootstrap ACK completion test
- missing master key test
- existing user / existing canonical sync identity test
- partial failure rollback test
- userId/username conflict validation tests
- concurrent create-if-missing race test
- operator auth failure test
- unknown-user offline/online regression tests
- OpenAPI coverage for the operator bootstrap surface

## Documentation Impact

The following docs should change when implementation lands:

- API reference
- administration guide
- developer guide
- OpenAPI / Swagger output

If the public base URL config is added, installation and deployment docs must
also be updated.

## Suggested Implementation Order

1. add explicit public base URL config
2. fix current operator/device correctness gaps
3. add a small bootstrap orchestration service
4. add `POST /admin/bootstrap/user-device`
5. add `POST /admin/bootstrap/{bootstrapRequestId}/ack`
6. emit operator auth + operator bootstrap in OpenAPI
7. add canonical sync identity and operator device lifecycle endpoints

## Success Criteria

The first issue 36 slice is successful when:

- a trusted remote operator workflow can bootstrap a user device without
  shelling into the admin CLI
- the operator API is emitted as a typed documented contract
- the implementation reuses narrow services rather than duplicating logic
- retries do not leave unusable partial state behind
- the resulting design is still explainable in ADR-0002 terms
